Project Overview and Objectives 1.1 Project Overview The Volume Link Discount Scheme (VLDS) Automation System is a multi-component software solution developed for Forest Dealerships to fully automate the quarterly incentive processing pipeline. The system replaces a manual, error-prone, three-to-five working day Excel-based process with a suite of integrated tools that deliver the complete incentive report in under fifteen minutes with zero manual calculation. The solution comprises four integrated components: Python Desktop Application — the core processing engine that ingests raw dealer sales data, applies nine sequential business logic steps, and generates a professionally formatted three-sheet Microsoft Excel incentive report with live Excel formulas. Microsoft Power Apps — two canvas applications providing structured form-based interfaces for Finance team members to submit dealer exclusion change requests and scheme parameter update requests. Microsoft Power Automate — automated approval workflows that route change requests through management approval, update SharePoint reference lists on approval, and notify stakeholders by email. SharePoint Online Portal — a centralised site serving as the document repository for scheme documentation, quarterly reference files, and distributed incentive reports. 1.2 Business Need Prior to this system, the Finance team processed VLDS incentive data entirely by hand using Microsoft Excel. This manual process required two to three analysts working for three to five working days per quarter and suffered from systematic deficiencies including inconsistent exclusion list application, dealer code format mismatches in previous year lookups, manual slab incentive rate entry errors, and no audit trail of processing decisions. Reduce quarterly processing time from three to five working days to under fifteen minutes. Deliver a fully dynamic system — all business rules are read from uploaded files at runtime with zero code changes required between quarters. Generate a standardised three-sheet Excel report with live Excel formulas enabling post-generation corrections without re-running the application. Provide a complete downloadable audit log for every processing run to support dealer dispute resolution and internal audit requirements. Deploy as a standalone Windows executable requiring no Python installation on client machines. Implement Power Apps forms and Power Automate workflows for governed scheme parameter change management. Centralise all scheme documentation and quarterly reports on a SharePoint portal. 2. Every row is scanned. The first numeric value in each cell is extracted using Python regex (re.findall). Header rows containing text like 'Quarterly', 'Growth %' are silently skipped. Decimal fraction payouts (0.02 meaning 2%) are auto-converted when detected. Phase 2 — Bracket Construction Parsed rows are sorted by growth lower bound ascending. Upper bound = next row's lower bound. Last row gets 999999 — catches all growth above the last threshold. Phase 3 — Nested IF Formula Generation _build_disc_formula() converts slab brackets into a nested Excel IF formula. For 4 brackets, the formula for row 4 is: =IF(H4<10.0, 0.5, IF(H4<20.0, 1.0, IF(H4<30.0, 1.5, 2.0))) Where H4 = Growth % cell in the same row. When user edits Q-prev in Excel: Increase recalculates → Growth % recalculates → Discount % (nested IF) recalculates → Discount Value recalculates 4. Logic and Business Rules Implemented 6.1 4S Dealer Eligibility (Step 2) Only dealers present in the 4S Dealer List are eligible. Matching is an inner join on normalised dealer code. Normalisation strips the '.0' decimal suffix from float-formatted codes (e.g., '12345.0' → '12345'). Dealers not in the list are removed and counted in the log. 6.2 LOB Eligibility (Step 3) Only PARTS and HIGHWAY Lines of Business are eligible. The LOB filter is case-insensitive. Rows with any other LOB value are removed. 6.3 Exclusion List (Step 4) Exclusion criteria are read dynamically from the uploaded Combined Reference File Sheet 2. Each criterion specifies a 'Particulars' value and a 'Column' identifier. For each criterion, matching rows in the VLDS data are removed. No hardcoded exclusion values exist in any source code file — the system is fully dynamic. 6.4 DF-RECD Matching (Step 5) The DF-RECD file identifies invoice records for exclusion. A composite BD key is constructed from multiple identifying columns. Both sides are normalised before comparison. Matched rows are removed from the eligible set. 6.5 Growth % Calculation (Step 8) Condition Formula Notes Previous Year > 0 Growth % = (Q-curr − Q-prev) / Q-prev × 100 Standard growth calculation Previous Year = 0 or dealer absent Growth % = 0 Division-by-zero protection Negative growth Growth % = negative value Preserved — falls into lowest slab bracket 6.6 Slab-Based Discount % (Step 8) Discount % is determined by looking up Growth % against sorted slab brackets using sequential scan: lo ≤ growth < hi returns that bracket's payout rate. If growth exceeds all bracket upper bounds, the last bracket's rate is returned. The same logic is also encoded as a nested IF Excel formula in the output, enabling live recalculation in Excel. 6.7 Discount Value Discount Value = Final Amount Q-curr × Discount % / 100. Assumptions and Limitations 9.1 Assumptions The VLDS Main File contains at least one column identifiable as Revenue by keyword matching. Fallback: last numeric column. Dealer codes may appear in different formats (integer vs float) — the system normalises both sides before matching. The Combined Reference File has at least three sheets; sheets map by keyword or position (1st=Dealer, 2nd=Exclusion, 3rd=Slab). Previous Year Summary is the output of a prior VLDS run for the same quarter of the previous year. Slab payouts stored as decimal fractions (e.g., 0.02) represent percentages (2%) — auto-converted when all values < 0.10. Microsoft Excel 2016 or later is required to open the output report and use the live formula functionality. Power Platform components require an active Microsoft 365 licence and internet connectivity. Single-user processing — no concurrent multi-user support. 9.2 Limitations Limitation Detail Workaround Discount % static after generation Nested IF formula reflects slab at generation time only Re-run with updated slab file for new formula Single-quarter processing No batch mode or annual aggregation Run four times; consolidate manually 200 MB file size limit Very large files may exceed this Split input files before processing Windows only macOS/Linux not supported in .exe format Run on Windows machine Prev year auto-detection Non-standard column names may cause wrong detection Verify auto-detected columns in UI info box before Run Power Platform needs internet Power Apps, Automate, SharePoint are cloud services Desktop app works fully offline Single-level approval Power Automate routes to one approver only Edit workflow to add stages if needed 10.
